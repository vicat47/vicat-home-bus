# ADR-0023: 使用 Apache Kafka 进行服务间通信

- **状态**: 已采纳
- **日期**: 2025-11-15
- **决策者**: 架构团队
- **相关文档**: [系统设计文档](docs/design.md)
- **受影响的 C4 元素**: [订单服务 (Container)](../c4/container-payment-service.md), [消息总线 (Container)](../c4/container-message-bus.md)

## 背景

我们的平台在过去一年从 3 个微服务增长到 12 个。
服务目前通过 HTTP REST 调用同步通信。这造成了几个问题：

- 级联故障：当库存服务宕机时，订单服务无法处理订单，
  即使库存检查可以是最终一致的。
- 紧耦合：服务需要知道彼此的 API 契约和端点。
- 性能瓶颈：某些操作触发 4-5 个同步调用链，增加了延迟。

我们每天处理大约 10 万个事件。预计 12 个月内增长到 50 万。
团队有 8 名后端工程师，其中 2 人有 Kafka 经验。

## 决策

我们将采用 Apache Kafka 作为异步服务间通信的主要机制。
同步 HTTP 将保留用于调用方需要即时结果的请求/响应模式
（例如认证检查）。

我们将使用 Confluent Cloud 作为托管 Kafka 提供商，
以最小化运维开销。

## 决策驱动因素

- 需要服务解耦和异步通信
- 需要事件重放能力用于审计和调试
- 预计未来 12 个月消息量增长 5 倍
- 希望避免加深 AWS 供应商锁定

## 考虑的选项

### 选项 1: Apache Kafka (Confluent Cloud)

**优点:**
- 基于日志的架构，强大的事件重放能力
- 高吞吐量，适合事件驱动架构
- 托管服务减少运维负担
- 社区活跃，生态成熟

**缺点:**
- 运维复杂性增加（Kafka 集群、Schema Registry、消费者组管理）
- 团队需要 Kafka 概念和模式的培训
- 异步流程中最终一致性替代强一致性

**拒绝/接受原因:**
- 接受：满足事件重放需求，适合预期的事件驱动架构方向

### 选项 2: RabbitMQ

**优点:**
- 更简单的运维，更低的学习曲线
- 支持多种消息模式（发布/订阅、点对点、路由）

**缺点:**
- 不太适合事件溯源模式
- 重放/回溯能力较弱
- 社区动力已转向 Kafka 用于事件驱动架构

**拒绝/接受原因:**
- 拒绝：预计需要事件重放用于审计和调试，Kafka 更适合

### 选项 3: AWS SQS + SNS

**优点:**
- 完全托管，无需维护基础设施
- 与 AWS 紧密集成

**缺点:**
- 对 AWS 的供应商锁定
- 消息排序保证有限
- 没有内置的流处理
- 在预期量级下每消息成本更高

**拒绝/接受原因:**
- 拒绝：希望避免加深 AWS 锁定，需要有序消息传递

## 影响

### 正面影响
- 服务解耦：生产者不需要知道消费者
- 异步工作流消除级联故障
- 事件重放实现强大的调试和审计能力
- 为未来服务的事件溯源模式奠定基础

### 负面影响/风险
- 运维复杂性增加（通过使用 Confluent Cloud 缓解）
- 异步流程需要重新设计某些工作流
- 调试分布式异步流程比同步 HTTP 调用更难
- 如果消息量超过定价层级，成本可能显著增加
- 跨服务的 Schema 演进需要纪律
